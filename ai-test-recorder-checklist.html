Tolerance for Flaky Tests
Why this matters

This checklist grew out of hands-on experience with AI test recorders on real projects — from simple forms to complex SPAs with shadow DOM, iframes, and third-party integrations. Each criterion reflects a practical concern, not a theoretical one.

I organized everything into 9 capability blocks with two priority tiers: Must (this is a must-have for launch) and Nice (what separates a good tool from an excellent one). Use it to evaluate vendors, guide internal tooling decisions, or benchmark your own recorder.

01
Foundation

Code Quality

Readable, maintainable code is the foundation everything else builds on.

Stable selector strategy
Must

Prioritize semantic locators: role → label → test-id → text. Never nth-child, never positional XPath, never auto-generated CSS-in-JS hashes.

Page Object Model adherence
Must

Generated tests should follow POM, Screenplay, or fixture-based architecture — not 200-line linear scripts.

AAA structure
Must

Arrange, Act, Assert visually and structurally separated within each test.

Readable naming
Must

Meaningful variable and function names. No magic numbers. Logical block separation.

Automatic helper extraction
Nice

Common steps like login or navigation extracted into reusable helpers, not duplicated across tests.

Project-level DRY
Nice

If ten tests share the same setup, the recorder proposes a fixture instead of cloning the code.

Specimen — Selector Strategy
What weak recorders generate
// Breaks on any DOM change
await page.click(
  'div.css-1a2b3c4 > span:nth-child(3) > button'
);

await page.click(
  '//*[@id="root"]/div/div[2]/form/button'
);
What a good recorder generates
// Survives refactors. Reads like intent.
await page
  .getByRole('button', { name: 'Submit order' })
  .click();

await page
  .getByTestId('checkout-confirm')
  .click();
02
Reliability

Waits & Async

The single largest source of flaky tests — and the area where recorders most need to get it right.

No waitForTimeout
Must

Hard-coded sleeps are an antipattern. They make tests slow when they pass and unreliable when they fail.

Element state waits
Must

toBeVisible, toBeEnabled, toHaveText — wait for readiness, not just presence.

Network response waits
Must

waitForResponse filtered by URL and status, before asserting downstream state.

URL & navigation waits
Must

Detect SPA route changes and full navigations correctly.

Custom condition waits
Nice

waitForFunction for non-trivial cases — animations, third-party widgets, timing-dependent UI.

Specimen — Waits & Async
Fragile: hard-coded sleep
// Slow when it works, flaky when it doesn't
await page.click('#submit');
await page.waitForTimeout(3000);
await expect(page.locator('.success'))
  .toBeVisible();
Robust: event-driven wait
// Waits only as long as needed
await page.click('#submit');
await page.waitForResponse(
  resp => resp.url().includes('/api/orders')
    && resp.status() === 200
);
await expect(page.locator('.success'))
  .toBeVisible();
03
Edge of the DOM

Complex Structures

Modern apps use iframes, shadow roots, and canvas — a good recorder should handle all of them.

Shadow DOM piercing
Must

Selectors must cross shadow root boundaries — critical for web components, Stripe Elements, design systems.

Iframe context switching
Must

Automatic frameLocator usage. The recorder should detect iframe entry without manual annotation.

Canvas & WebGL interaction
Nice

At minimum, coordinate-based interaction for maps, charts, editors, design tools.

Virtualized lists
Nice

React-window, virtual scroll, infinite lists — items not in DOM until scrolled.

04
Inputs

User Interaction

Beyond clicks — keyboard, uploads, dialogs, and multi-tab scenarios matter just as much.

Keyboard handling
Must

Hotkeys, key combinations (Ctrl+Shift+...), sequential input, Tab navigation, Escape, Enter.

Modals & native dialogs
Must

Custom modals plus native browser alert, confirm, prompt — different mechanics, both required.

Toasts & flash messages
Must

Catch and assert ephemeral notifications before they auto-dismiss (typically 3–5 seconds).

File upload & download
Must

Verify downloaded file name, size, and content. Handle multi-file uploads and drag-to-upload zones.

Clipboard operations
Nice

Read and write clipboard content for copy-paste flows.

Multiple browser contexts
Must

Multi-user scenarios (chat, collaboration), new tabs, incognito sessions, isolated cookies.

05
State

Test Data

Bad data handling produces tests that conflict, cascade-fail, and can't run in parallel.

Centralized authentication
Must

storageState or equivalent — never log in through the UI in every test.

Data-driven parameterization
Must

One scenario, multiple data sets. Avoid copy-pasting the same test with different inputs.

Unique data generation
Must

Timestamps, faker, UUIDs — so parallel runs don't collide on the same email or order ID.

Scoped fixtures
Nice

Test, worker, and global scope — match data lifecycle to test lifecycle.

Setup & teardown via API
Must

Faster and more reliable than reaching the same state through the UI.

Automatic cleanup
Must

Hooks or isolation strategies that prevent test pollution between runs.

Fixture import & export
Nice

Load test data from JSON/CSV, export state snapshots for debugging.

Specimen — Authentication & Data
Slow: UI login in every test
// Repeated in every single test file
test('create order', async ({ page }) => {
  await page.goto('/login');
  await page.fill('#email', 'admin@test.com');
  await page.fill('#password', 'password123');
  await page.click('#login-btn');
  // ... actual test starts here
});
Fast: stored auth state + unique data
// Auth once, reuse across all tests
test.use({
  storageState: 'auth/admin.json'
});

test('create order', async ({ page }) => {
  const orderId = randomUUID();
  await page.goto(`/orders/new?ref=${orderId}`);
  // No login, no collisions in parallel
});
06
Boundaries

Network & Mocks

Controlling network boundaries makes tests faster, more predictable, and independent of backend availability.

Network request interception
Must

page.route for selectively mocking endpoints while letting others pass through.

Error response mocking
Must

500s, timeouts, empty responses, malformed payloads — edge cases you can't reproduce on staging.

Time, geolocation, permissions
Nice

Mock browser APIs for deterministic tests of time-sensitive or location-aware features.

API calls from tests
Must

For setup, teardown, and verifying backend state without going through the UI.

External test service integration
Nice

Mailcatcher, Mailhog, SMS interceptors, webhook receivers — verify side effects outside the app.

Specimen — Network Mocking
Untestable: depends on live backend
// Fails if staging is down or data changes
await page.goto('/dashboard');
await expect(
  page.locator('.revenue')
).toHaveText('$12,340');
// What if the amount changes tomorrow?
Controlled: mock the API response
await page.route('**/api/revenue', route =>
  route.fulfill({
    json: { total: 12340, currency: 'USD' }
  })
);

await page.goto('/dashboard');
await expect(
  page.locator('.revenue')
).toHaveText('$12,340');
07
Logic

Flow & Assertions

Recorders capture a single happy path by default — the question is how gracefully they handle everything that isn't one.

Conditional logic
Nice

If/else in scenarios — for example, dismissing a cookie banner only when it appears. Most recorders capture a single path; branching is typically added by hand.

Loops
Nice

Iterate over lists, repeated UI elements, dynamic table rows.

Try/catch for optional steps
Nice

Gracefully handle steps that may or may not be present depending on user state.

Meaningful assertions
Must

Not just "element is visible." Assert text content, attribute values, element counts, computed states. The recorder should propose what to verify, based on context.

Soft assertions
Nice

Collect multiple checks in one test without failing on the first mismatch.

Specimen — Assertions
Weak: only checks visibility
// Passes even if the content is wrong
await expect(
  page.locator('.order-confirmation')
).toBeVisible();
Strong: verifies actual state
const confirmation = page.locator(
  '.order-confirmation'
);
await expect(confirmation)
  .toContainText('Order #');
await expect(confirmation.locator('.status'))
  .toHaveText('Confirmed');
await expect(confirmation.locator('.items'))
  .toHaveCount(3);
08
Operations

Execution & Scale

Most execution concerns live in the runner and CI config. What matters here is that the generated code plays well with them.

Test isolation
Must

Each test must be independent. No shared state, no execution order dependencies, no leftover data from previous runs.

Cross-browser compatibility
Must

Generated code uses APIs and selectors that work across Chromium, Firefox, and WebKit — not browser-specific quirks.

Responsive & device emulation
Nice

Support for viewport switching and mobile device profiles when recording — not just desktop-only capture.

09
Responsibility

Security & Privacy

Test recordings capture everything. That's a feature and a liability.

Secrets management
Must

Integration with vaults and environment variables. Credentials never in plain text.

GDPR & data residency
Must

Region-specific data storage options for compliance with EU and other regulations.

Red Flags

Signs worth paying attention to.

Demo runs only on a static landing page

Real apps have async loading, modals, and auth. It's important to see how the tool performs on your actual application, not just a simplified example.

Generated code uses nth-child or absolute XPath

This is a fundamental design failure. The tool will produce flaky tests forever.

No way to inspect or edit the generated code

Pure visual editors mean you can't debug, refactor, or escape the platform.
